# 브랜치 전략

> Midnight Post Office 프로젝트의 Git 브랜치 운영 규칙입니다.

---

## 브랜치 구조

```
main
 └── 항상 안정적인 기획/문서 상태 (현재)
     → Phase 1 시작 후부터는 배포 가능한 코드만 유지

release
 └── 배포 버전 기록 전용
     → develop에서 검증 완료된 버전만 머지
     → v0.1.0, v0.2.0 등 태그로 버전 관리

develop
 └── 기능 통합 브랜치 (현재 기본 작업 브랜치)
     → feature/* 브랜치들이 여기로 머지
     → 전체 기능 조합 후 테스트

feature/*
 └── 기능 단위 개발 브랜치
     → AI가 기능 작업 시 여기서 작업
     → 완료 후 develop으로 PR

fix/*
 └── 버그 수정 브랜치
     → develop 또는 main 기반으로 생성
     → 수정 후 develop으로 PR
```

---

## 브랜치별 역할 상세

### `main`
- 언제나 **동작 가능한 안정 상태**
- 직접 커밋 금지 — release에서만 머지
- 현재: 기획 문서 보관

### `develop`
- **기본 작업 브랜치**
- 모든 feature 브랜치의 통합 대상
- feature 머지 후 전체 동작 테스트 수행
- 불안정해도 괜찮음 (테스트 용도)

### `release`
- **배포 버전 기록 전용**
- develop에서 충분히 검증된 후에만 머지
- 머지 시 반드시 버전 태그 생성
  ```bash
  git tag -a v0.1.0 -m "Phase 1 MVP 완료"
  git push origin v0.1.0
  ```

### `feature/*`
- 기능 단위로 생성
- 아래 네이밍 규칙 참고

## 작업 흐름 요약

```
1. AI가 feature/* 브랜치에서 코드 작성
        ↓
2. 사람이 코드 검토 후 develop에 머지 (PR)
        ↓
3. develop에서 전체 기능 통합 테스트
        ↓
4. 안정화 확인 후 release에 머지
        ↓
5. 버전 태그 생성 (v0.x.0)
        ↓
6. release → main 머지 (배포 시점)
```

---

## AI에게 작업 요청하는 방법

특정 기능 개발을 요청할 때:

```
feature/p1-auth 브랜치에서 Google 로그인 구현해줘.
```

AI는 자동으로 해당 브랜치를 체크아웃하고 작업합니다.
