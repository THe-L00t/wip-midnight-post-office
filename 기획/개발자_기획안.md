# 2. 핵심 게임 로직 설계

## 편지 전달의 기본 원칙

- 편지는 **자동으로 전달되지 않는다.**
- 편지가 중간 우체국에 도착하면, 해당 우체국 주인이 **직접 접속하여 방문 전달**해야 다음 경유지로 이동한다.
- 방문은 **관계가 형성된 유저의 우체국**에만 가능하다.
- 관계는 두 유저 사이에 **편지가 3번 이상 오고 간 경우** 자동 형성된다.

## 관계 형성 흐름

```
편지 교환 1회 → 2회 → 3회 이상
                          ↓
               관계 형성 (relationship 생성)
                          ↓
          상대 우체국 방문 가능
          편지 라우팅 경로로 활용 가능
```

```sql
CREATE TABLE post_offices (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID REFERENCES users(id) UNIQUE,
  theme       TEXT DEFAULT 'classic',
  lighting    TEXT DEFAULT 'warm',
  music_track TEXT,
  decoration  JSONB DEFAULT '{}',
  visit_count INTEGER DEFAULT 0,
  created_at  TIMESTAMPTZ DEFAULT now()

```

### letter_exchanges (편지 교환 횟수 추적 — 관계 형성 판정용)
```sql
CREATE TABLE letter_exchanges (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_a_id    UUID REFERENCES users(id),
  user_b_id    UUID REFERENCES users(id),
  exchange_count INTEGER DEFAULT 1,   -- 편지 오고 간 횟수
  updated_at   TIMESTAMPTZ DEFAULT now(),
  UNIQUE (user_a_id, user_b_id)       -- user_a < user_b 정렬 저장
);
```

> `user_a_id < user_b_id` 순서로 항상 저장하여 중복 방지.
> 3 이상이 되는 순간 `relationships` 테이블에 관계 자동 생성.

### relationships (관계망 — 방문 가능 + 라우팅 경로)
```sql
CREATE TABLE relationships (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  from_user_id UUID REFERENCES users(id),
  to_user_id   UUID REFERENCES users(id),
  strength     INTEGER DEFAULT 1,     -- 지속적 교류로 증가
  formed_at    TIMESTAMPTZ DEFAULT now(),
  UNIQUE (from_user_id, to_user_id)
);

-- 양방향 탐색용 인덱스
CREATE INDEX idx_rel_from ON relationships(from_user_id);
CREATE INDEX idx_rel_to   ON relationships(to_user_id);
```

> 관계는 **양방향**으로 두 행 삽입 (A→B, B→A).
> 관계가 있어야 상대 우체국 방문 및 편지 경유 경로 사용 가능.

### letters (편지 — 현재 대기 위치 포함)
```sql
CREATE TABLE letters (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  sender_id       UUID REFERENCES users(id),
  recipient_id    UUID REFERENCES users(id),
  content         TEXT NOT NULL,
  stamp_type      TEXT,
  status          TEXT DEFAULT 'waiting',
                      -- waiting     : 현재 경유지에서 대기 중 (유저 전달 필요)
                      -- in_transit  : 다음 경유지로 이동 완료 (다음 유저 대기)
                      -- delivered   : 최종 수신자에게 도달
  current_holder  UUID REFERENCES users(id),  -- 현재 편지를 갖고 있는 우체국 주인
  route           UUID[],                      -- 전달 경로 배열 [sender, B, D, recipient]
  hop_index       INTEGER DEFAULT 0,           -- 현재 몇 번째 경유인지 (0 = 발신자)
  created_at      TIMESTAMPTZ DEFAULT now(),
  delivered_at    TIMESTAMPTZ
);
```

> `current_holder`: 편지가 현재 누구의 우체국에 있는지를 나타냄.
> 이 유저가 접속해서 다음 우체국을 방문해야만 `hop_index`가 증가한다.

### stamps (우표 컬렉션)
```sql
CREATE TABLE stamps (
```

```sql
CREATE TABLE unlocks (
```

# 4. 핵심 시스템 구현

## 4-1. 편지 라우팅 경로 탐색 (최초 1회)

편지를 처음 보낼 때, **관계망을 기반으로 최단 경로를 탐색**한다.
경로는 서버(Edge Function)에서 계산하고 `letters.route`에 저장한다.

```sql
-- sender에서 recipient까지 관계망 기반 BFS 최단 경로 탐색
-- 조건: relationships 테이블에 존재하는 연결(관계 형성된 유저)만 사용
WITH RECURSIVE path AS (
  SELECT
    from_user_id,
    to_user_id,
    ARRAY[from_user_id, to_user_id] AS route,
    1 AS depth
  FROM relationships
  WHERE from_user_id = :sender_id

  UNION ALL

  SELECT
    r.from_user_id,
    r.to_user_id,
    p.route || r.to_user_id,
    p.depth + 1
  FROM relationships r
  JOIN path p ON r.from_user_id = p.to_user_id
  WHERE r.to_user_id != ALL(p.route)
    AND p.depth < 10
)
SELECT route
FROM path
WHERE to_user_id = :recipient_id
ORDER BY depth ASC
LIMIT 1;
```

> 경로를 찾을 수 없는 경우: 유저에게 "아직 이 사람에게 닿는 길이 없어요" 안내.

## 4-2. 편지 직접 전달 (유저 액션)

유저가 관계 있는 우체국을 방문해 "전달하기"를 누를 때 실행되는 로직.

```typescript
// supabase/functions/relay-letter/index.ts
serve(async (req) => {
  const { letter_id, actor_user_id } = await req.json();
  const supabase = createClient(...);

  // 1. 편지 조회
  const { data: letter } = await supabase
    .from('letters')
    .select('*')
    .eq('id', letter_id)
    .single();

  // 2. 액션 유저가 현재 편지 보유자인지 검증
  if (letter.current_holder !== actor_user_id) {
    return new Response('Unauthorized', { status: 403 });
  }

  // 3. 다음 경유지 계산
  const nextHopIndex = letter.hop_index + 1;
  const nextHolder   = letter.route[nextHopIndex];
  const isDelivered  = nextHopIndex === letter.route.length - 1;

  // 4. 편지 상태 업데이트
  await supabase.from('letters').update({
    hop_index:      nextHopIndex,
    current_holder: nextHolder,
    status:         isDelivered ? 'delivered' : 'waiting',
    delivered_at:   isDelivered ? new Date().toISOString() : null,
  }).eq('id', letter_id);

  // 5. 경유 기록 저장
  await supabase.from('letter_logs').insert({
    letter_id,
    from_user_id: actor_user_id,
    to_user_id:   nextHolder,
  });

  // 6. 경유 보상 지급
  await rewardRelay(supabase, actor_user_id);

  // 7. 편지 교환 횟수 업데이트 → 관계 형성 판정
  await updateExchangeCount(supabase, letter.sender_id, letter.recipient_id);

  // 8. 다음 보유자에게 FCM 푸시 알림
  await sendPushNotification(nextHolder, isDelivered
    ? '편지가 도착했어요'
    : '당신의 우체국에 편지가 왔어요');

  return new Response('ok');
});
```

## 4-3. 관계 형성 자동 판정

편지 전달이 완료될 때마다 발신자-수신자 간 교환 횟수를 갱신하고,
3회 이상이 되면 관계를 자동 생성한다.

```sql
-- 편지 교환 횟수 upsert + 관계 형성 판정 함수
CREATE OR REPLACE FUNCTION update_exchange_and_form_relationship(
  p_sender_id    UUID,
  p_recipient_id UUID
) RETURNS VOID AS $$
DECLARE
  v_user_a      UUID;
  v_user_b      UUID;
  v_count       INTEGER;
BEGIN
  -- 항상 user_a < user_b 순서로 정렬 저장
  v_user_a := LEAST(p_sender_id, p_recipient_id);
  v_user_b := GREATEST(p_sender_id, p_recipient_id);

  -- 교환 횟수 증가
  INSERT INTO letter_exchanges (user_a_id, user_b_id, exchange_count)
  VALUES (v_user_a, v_user_b, 1)
  ON CONFLICT (user_a_id, user_b_id)
  DO UPDATE SET
    exchange_count = letter_exchanges.exchange_count + 1,
    updated_at = now()
  RETURNING exchange_count INTO v_count;

  -- 3회 이상이면 양방향 관계 생성
  IF v_count >= 3 THEN
    INSERT INTO relationships (from_user_id, to_user_id)
    VALUES (v_user_a, v_user_b)
    ON CONFLICT DO NOTHING;

    INSERT INTO relationships (from_user_id, to_user_id)
    VALUES (v_user_b, v_user_a)
    ON CONFLICT DO NOTHING;
  END IF;
END;
$$ LANGUAGE plpgsql;
```

## 4-4. 우체국 방문 가능 여부 검증

클라이언트에서 방문 요청 시 서버에서 관계 여부를 반드시 검증한다.

```sql
-- 방문 가능 여부 확인 (RLS 정책으로도 적용)
SELECT EXISTS (
  SELECT 1 FROM relationships
  WHERE from_user_id = :visitor_id
    AND to_user_id   = :target_user_id
) AS can_visit;
```

RLS(Row Level Security) 정책 적용:

```sql
-- post_offices: 관계가 있는 유저의 우체국만 SELECT 가능
CREATE POLICY "visit_only_connected"
ON post_offices FOR SELECT
USING (
  user_id = auth.uid()  -- 본인
  OR EXISTS (
    SELECT 1 FROM relationships
    WHERE from_user_id = auth.uid()
      AND to_user_id   = post_offices.user_id
  )
);
```

## 4-5. 중앙 우체국 시스템

### 개요

중앙 우체국은 시스템이 직접 운영하는 단일 공공 공간으로,
관계가 없는 유저 간 편지 전달의 시작점 역할을 한다.
개인 우체국과 달리 **모든 유저가 방문 가능**하다.

---

### 테이블 설계

#### central_post_office_letters (중앙 우체국 등록 편지)
```sql
CREATE TABLE central_post_office_letters (
```

```sql
  user_id    UUID REFERENCES users(id),
```

### 이용 횟수 제한 로직 (Edge Function)

```typescript
// supabase/functions/register-to-central/index.ts
serve(async (req) => {
  const { letter_id, user_id } = await req.json();
  const supabase = createClient(...);

  const yearMonth = new Date().toISOString().slice(0, 7); // 'YYYY-MM'

  // 1. 이번 달 이용 횟수 조회
  const { data: usage } = await supabase
    .from('central_post_office_usage')
    .select('used_count')
    .eq('user_id', user_id)
    .eq('year_month', yearMonth)
    .maybeSingle();

  const usedCount = usage?.used_count ?? 0;
  const MONTHLY_LIMIT = 3;

  // 2. 제한 초과 여부 검사
  if (usedCount >= MONTHLY_LIMIT) {
    return new Response(
      JSON.stringify({ error: 'monthly_limit_exceeded', used: usedCount, limit: MONTHLY_LIMIT }),
      { status: 429 }
    );
  }

  // 3. 중앙 우체국에 편지 등록
  await supabase.from('central_post_office_letters').insert({
    letter_id,
    registered_by: user_id,
  });

  // 4. 이용 횟수 증가 (upsert)
  await supabase.from('central_post_office_usage').upsert({
    user_id,
    year_month: yearMonth,
    used_count: usedCount + 1,
  }, { onConflict: 'user_id, year_month' });

  return new Response(JSON.stringify({ ok: true, remaining: MONTHLY_LIMIT - usedCount - 1 }));
});
```

### 편지 가져가기 (픽업)

```typescript
// supabase/functions/pickup-from-central/index.ts
serve(async (req) => {
  const { central_letter_id, picker_user_id } = await req.json();
  const supabase = createClient(...);

  // 1. 아직 픽업되지 않은 편지인지 확인
  const { data: entry } = await supabase
    .from('central_post_office_letters')
    .select('*, letters(*)')
    .eq('id', central_letter_id)
    .is('picked_up_by', null)
    .single();

  if (!entry) {
    return new Response('Already picked up or not found', { status: 404 });
  }

  // 2. 자기 자신이 등록한 편지는 픽업 불가
  if (entry.registered_by === picker_user_id) {
    return new Response('Cannot pick up your own letter', { status: 403 });
  }

  // 3. 픽업 처리: 편지의 current_holder를 픽업한 유저로 전환
  await supabase.from('central_post_office_letters').update({
    picked_up_by: picker_user_id,
    picked_up_at: new Date().toISOString(),
  }).eq('id', central_letter_id);

  await supabase.from('letters').update({
    current_holder: picker_user_id,
    status: 'waiting',
  }).eq('id', entry.letter_id);

  // 4. 픽업한 유저에게 새 경로 탐색 위임 (픽업 후 자신의 관계망으로 전달)
  return new Response(JSON.stringify({ ok: true, letter: entry.letters }));
});
```

```dart
// Flutter: 이번 달 중앙 우체국 잔여 횟수 표시
Future<int> getCentralUsageRemaining(String userId) async {
  final yearMonth = DateTime.now().toIso8601String().substring(0, 7);
  final res = await supabase
    .from('central_post_office_usage')
    .select('used_count')
    .eq('user_id', userId)
```

### 월 제한 설계 의도

| 항목 | 값 | 이유 |
|------|----|------|
| 월 기본 제한 | 3회 | 초반 네트워크 진입은 허용하되, 의존 방지 |
| 추가 이용 | 재화 소모 1회 | 완전 차단 대신 비용을 통한 자연 억제 |
| 재화 없이 해제 | 다음 달 리셋 | 시간이 지나면 자연 회복 |

> 제한의 목적은 차단이 아니라 **개인 관계망 형성을 자연스럽게 유도**하는 것.
> 중앙 우체국을 거친 편지도 편지 교환 횟수에 포함되어 관계 형성 조건(3회)에 기여한다.

## 4-6. 경유 보상 처리

```typescript
async function rewardRelay(supabase, user_id: string) {
  // 명성 +1
  await supabase.rpc('increment_reputation', { user_id, amount: 1 });

  // 우표 랜덤 지급 (10% 확률)
  if (Math.random() < 0.1) {
    await supabase.from('stamps').insert({
      user_id,
      stamp_type: pickRandomStamp(),
      source: 'relay_reward',
    });
  }
}
```

## 4-6. 관계 해금 시스템

관계 강도(`relationships.strength`)가 임계값을 넘을 때 해금 항목 자동 생성.

```sql
CREATE OR REPLACE FUNCTION check_unlock_on_strength(
  p_user_a UUID, p_user_b UUID, p_strength INTEGER
) RETURNS VOID AS $$
BEGIN
  IF p_strength = 10 THEN
    INSERT INTO unlocks (user_a_id, user_b_id, unlock_type)
    VALUES (p_user_a, p_user_b, 'joint_stamp')
    ON CONFLICT DO NOTHING;
  END IF;

  IF p_strength = 30 THEN
    INSERT INTO unlocks (user_a_id, user_b_id, unlock_type)
    VALUES (p_user_a, p_user_b, 'secret_music')
    ON CONFLICT DO NOTHING;
  END IF;

  IF p_strength = 50 THEN
    INSERT INTO unlocks (user_a_id, user_b_id, unlock_type)
    VALUES (p_user_a, p_user_b, 'private_space')
    ON CONFLICT DO NOTHING;
  END IF;
END;
$$ LANGUAGE plpgsql;
```

## 추천 스택 구성

```
[Flutter 앱 — Android / iOS / Windows]
          ↓
    [Supabase]
      ├─ PostgreSQL     ← 모든 도메인 데이터 + 관계망 + 편지 상태
      ├─ Auth           ← 소셜 로그인 (Google, Apple)
      ├─ Realtime       ← 편지 대기 알림 실시간 구독
      ├─ Storage        ← 이미지, 음악, 우표 에셋
      └─ Edge Functions ← 편지 라우팅, 전달 처리, 관계 형성 판정
```

## 서비스별 선택 근거

### Supabase
- PostgreSQL 기반 관계망, 편지 상태, 교환 횟수 단일 관리
- `$25/월 고정` → 비용 예측 가능
- RLS로 방문 권한 제어 (관계 없는 우체국 접근 차단)
- 오픈소스, 벤더 락인 없음

### Firebase FCM
- 완전 무료, 메시지 수 제한 없음
- `firebase_messaging` Flutter 공식 지원
- "편지가 당신의 우체국에 도착했어요" 등 감성 알림 처리

# 10. 보안 고려사항

- **RLS (Row Level Security)**: 관계 없는 유저의 우체국 데이터 접근 차단
- **편지 전달 권한 검증**: `current_holder === actor_user_id` 서버에서 반드시 검증 (클라이언트 신뢰 금지)
- **경로 조작 방지**: 편지 라우팅 경로(`route` 배열)는 서버(Edge Function)에서만 결정
- **스팸 방지**: 일일 편지 발송 횟수 Rate Limit 설정
- **교환 횟수 조작 방지**: `letter_exchanges` 업데이트는 Edge Function 내부에서만 수행
